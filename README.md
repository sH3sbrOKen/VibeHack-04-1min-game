# 1min-game

> **AI 开发者请先读完本文件，再根据文件目录找具体文档。**

## 项目概述

**游戏名备选**：「1%」/「99+」/「今天不想回消息」
**类型**：1分钟休闲解谜，手机竖屏，纯网页实现（单个 HTML 文件，无框架）
**核心机制**：手机1%电，99+消息，四角预设回复，答错触发隐藏数值，数值解锁支线关卡
**开发方式**：无代码，全程AI生成（HTML + JavaScript）
**黑客松**：VibeFriends x 张江AI创新小镇 x Aseed+ / 24h Vibe Coding / 2026-05-29~30 上海

## 核心概念速查

### 四个回复按钮

| 位置 | ReplyType | 文字 | 颜色 | 适用场景 |
|------|-----------|------|------|--------|
| 左上 | WAIT | 等等 | 橙 #FF9500 | 需要物理动作（开门禁、下楼取件） |
| 右上 | RECEIVED | 收到 | 蓝 #007AFF | 确认知悉（老板、系统通知） |
| 左下 | UNSUBSCRIBE | 退订 | 红 #FF3B30 | 骚扰、诈骗、营销 |
| 右下 | LOVE | 好的亲 | 绿 #34C759 | 情感类（对象、家人） |

### 正确答案推导

优先级：**正文 BodyOverrideRule > 发信人 defaultReply**

```
RULES 数组逐条匹配正文 → 命中返回 override
未命中 → 返回 DEFAULTS[senderType]
```

### 六个隐藏数值 → 解锁支线关卡

| StatKey | 初始 | 触发阈值 | 方向 | 解锁关卡 |
|---------|------|--------|------|--------|
| DELIVERY_LOST | 0 | ≥3 | 越高越坏 | 骑手一分钟 |
| BOSS_PATIENCE | 100 | ≤0 | 越低越坏 | 职场危机一分钟 |
| INFO_LEAK | 0 | ≥3 | 越高越坏 | 电话销售一分钟 |
| INTIMACY | 100 | ≤0 | 越低越坏 | 分手现场一分钟 |
| SPAM_SUBSCRIBED | 0 | ≥5 | 越高越坏 | 骚扰接线员一分钟 |
| UNEXPECTED_CHARGE | 0 | ≥200 | 越高越坏 | 客服投诉一分钟 |

### 信息卡字段

```js
{ t: SenderType, n: '显示名', b: '正文', e: {StatKey: delta}, d: 1|2|3, trap?: true, easter?: true }
```

## index.html 内部结构

文件分三层，修改时只动对应层：

```
<style>   ← 视觉层：配色、布局、动画样式
<body>    ← 结构层：HTML骨架、按钮、弹窗
<script>  ← 逻辑层：游戏规则、数值、消息判断
```

- **视觉改动**：只动 `<style>`，不碰 `<script>`
- **内容改动**：只动 `<script>` 里的 `RAW` 数组
- **规则改动**：只动 `<script>` 里的 `RULES` 数组和 `STAT_CFG`
- **动画改动**：`@keyframes` 在 `<style>` 里，但命名被 JS 引用，改名要同步

## 游戏流程

```
开始画面 → startGame()
  ↓
nextMsg() 弹出信息卡（难度1→2→3分阶段）
  ↓
reply(r) 玩家点击四角按钮
  ├─ 答对 → cardPop 动画，继续
  ├─ 答错 → 触发 StatEffect，飘字反馈，数值条首次显现
  └─ 回完99条 → showCongrats()（极稀有彩蛋）→ dismissCongrats() → endGame()
  ↓
60秒耗尽 → endGame()
  ↓
结算页：展示数值 + 解锁提示 + 分享文案
```

## AI 开发规范

### 需求类型识别（最高优先级，每次收到需求必做）

收到用户输入时，先判断类型再行动：

**体验目标**：描述玩家应该感受到什么（"让玩家有压迫感"、"希望陷阱卡更难识别"）
- 必须先输出实现方案计划：做什么、改哪里、为什么这样实现
- 检查是否和现有功能冲突，如有冲突须在计划中说明
- 等用户确认后才能执行，不得直接开始改代码

**功能需求**：描述具体要做什么（"加一个震屏效果"、"把按钮改成方形"）
- 必须先说明：这个改动对游戏体验有什么影响（正面/负面/中性）
- 是否和现有功能有冲突或联动
- 说明完后可直接执行，无需用户二次确认

> 判断不确定时，按体验目标处理（先给方案等确认）。

---

### 开始新任务前必做

1. 读本文件确认最新文件状态
2. 读 `docs/dev/VERSIONS.md` 确认当前开发焦点和等待的设计决策
3. 明确本次任务影响哪一层（视觉/内容/逻辑）
4. `[DEV]` 任务直接开始；非 `[DEV]` 任务先说计划等确认

### 修改代码

- 只改任务涉及的最小范围，不顺带重构
- 视觉和逻辑分离，不交叉修改
- 每次改完说明改了哪里、为什么这样改

### 文档同步（开发完成后必做）

- 改完 `index.html` → 更新 `docs/dev/VERSIONS.md` 对应条目打 ✅
- 游戏规则有变化 → 同步 `docs/design/GDD.md`
- 新增/删除信息卡 → 同步 `docs/design/CONTENT.md`
- 文案有变化 → 检查是否符合 `docs/design/COPY-STYLE.md`

### 新增功能

- 新增 HTML 元素放在对应注释块附近
- 新增 CSS 放在对应模块注释下方
- 新增 JS 函数放在功能相关函数附近，保持分区注释结构

### 遇到问题

- 同一问题尝试3次无进展：停下来，描述卡点，问用户
- 逻辑和视觉冲突：优先保逻辑，视觉妥协
- 不确定需求时：列出理解方式，让用户选
- 技术方案有多种时：说明各方案优劣，推荐一个，让用户拍板

### 禁止事项

- 不重构运行正常的代码
- 不添加用户未要求的功能
- 不改动无关的 CSS 或 JS
- 不把单个 HTML 拆分成多文件（黑客松分发需要单文件）
- 不引入任何框架或构建工具
- 不自行做设计决策（数值、文案、规则改动必须先问用户）
