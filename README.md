# 1min-game

> **每次开发任务开始前：读完本文件 + `docs/dev/VERSIONS.md`**

手机1%电，99+消息，四角预设回复，答错触发隐藏数值，数值解锁支线关卡。
单个 HTML 文件，无框架，浏览器直接运行。黑客松项目，2026-05-29~30 上海。

---

## 文件地图

| 文件 | 用途 |
|------|------|
| `index.html` | **唯一可运行文件**，改代码只动这里 |
| `docs/design/GDD.md` | 游戏设计：核心循环、数值系统、支线关卡 |
| `docs/design/CONTENT.md` | 信息卡数据库：Schema、手写卡、规则引擎 |
| `docs/design/COPY-STYLE.md` | 文案风格规范 |
| `docs/dev/VERSIONS.md` | 版本历史 + 当前迭代计划（每次必读） |
| `docs/dev/ACCIDENT-LOG.md` | 开发事故日志 + 提炼规律 |
| `docs/dev/SELF-CHECK.md` | 功能自检清单（从事故日志提炼） |

**数据以 `index.html` 的 RAW 数组为准**，CONTENT.md 是设计稿。

## 协作规范

### 分工
- **设计**（用户）：游戏规则、数值、信息卡内容、文案、视觉方向、支线剧情
- **开发**（AI）：代码实现、bug修复、文档同步、技术可行性判断

设计输入只需说清楚：做什么、为什么。不需要说怎么实现——技术方案由AI决定，有更好方案或不可行时说明并等用户拍板。

### 需求类型识别（每次收到需求先判断）

**体验目标**（玩家应该感受到什么）→ 先出实现方案+冲突检查，等用户确认再执行

**功能需求**（具体做什么）→ 先说明对体验的影响和潜在冲突，说完直接执行

**视觉需求**（改样式/动画）→ 功能实现和视觉提升必须单独计划执行，不得混在一次改动里

未注明类型时必须反问确认。判断不确定时按体验目标处理（先给方案等确认）。

### 开发流程

1. 读本文件 + VERSIONS.md 确认当前焦点
2. 判断需求类型（见上）
3. 只改任务涉及的最小范围，视觉和逻辑不交叉
4. **功能完成后必须执行自检**（见下），报告结果
5. 改完同步文档（见文档同步规则）

### 开发事故与自检

**开发事故**：试玩中发现的体验问题。每次修正必须在 `ACCIDENT-LOG.md` 记录事故 + 提炼规律，并更新 `SELF-CHECK.md`。

**自检触发**：① 每次完成新功能后自动执行；② 用户说「自检」或「复盘」时执行。

自检方法：逐条过 `SELF-CHECK.md` 的检测维度，输出格式：
```
自检完成：[功能名]
SC-001 因果自洽：✅/⚠️/❌ [说明]
结论：通过 / 发现 X 个问题：[列出]
```

### 文档同步

- 改 `index.html` → 更新 `VERSIONS.md` 对应条目打 ✅
- 规则/数值变化 → 同步 `GDD.md`
- 信息卡增删 → 同步 `CONTENT.md`
- 修改任何指导性 MD 文件 → **整体重写而非追加条目**，保持长度适合AI单次阅读

### Git 规范

只在用户明确要求时执行 commit/push，不主动提示。格式：`type: 简述`

### 禁止事项

- 不重构运行正常的代码，不添加未要求的功能
- 不把 HTML 拆成多文件，不引入框架或构建工具
- 不自行做设计决策（数值/文案/规则改动必须先问用户）
- 视觉和逻辑改动不混在同一次提交

## 核心概念速查

### 四个回复按钮

| 位置 | 文字 | ReplyType | 适用场景 |
|------|------|-----------|--------|
| 左上 | 等等 | WAIT | 需要物理动作（开门禁、下楼取件） |
| 右上 | 收到 | RECEIVED | 确认知悉（老板指令、系统通知） |
| 左下 | 退订 | UNSUBSCRIBE | 骚扰、诈骗、营销 |
| 右下 | 好的亲 | LOVE | 情感类（对象、家人、朋友） |

### 正确答案推导

`RULES 数组逐条匹配正文 → 命中返回 override → 未命中返回 DEFAULTS[senderType]`

### 七个数值

| StatKey | 初始 | 阈值 | 解锁支线 |
|---------|------|------|--------|
| DELIVERY_LOST | 0 | ≥3 | 骑手一分钟 |
| BOSS_PATIENCE | 100 | ≤0 | 职场危机一分钟 |
| INFO_LEAK | 0 | ≥3 | 电话销售一分钟 |
| INTIMACY | 100 | ≤0 | 分手现场一分钟 |
| SPAM_SUBSCRIBED | 0 | ≥5 | 骚扰接线员一分钟 |
| UNEXPECTED_CHARGE | 0 | ≥200 | 客服投诉一分钟 |
| HANDLED（正向）| 0 | 无阈值 | 答对+1，结算展示 |

### 信息卡字段

```js
{ t: SenderType, n: '显示名', b: '正文', e: {StatKey: delta}, d: 1|2|3, trap?: true, easter?: true }
```

wrongEffect 赋值原则：**玩家回错后在现实中必须能因果造成该效果**（见 SC-001）。

### index.html 三层结构

```
<style>   视觉层 — 配色、布局、动画
<body>    结构层 — HTML骨架、按钮、弹窗
<script>  逻辑层 — 游戏规则、数值、消息判断
```

视觉改动只动 `<style>`，逻辑改动只动 `<script>`，不交叉。
